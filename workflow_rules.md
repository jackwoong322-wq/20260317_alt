## 이슈 처리 사이클 상세

### STEP 1 — 분석 & 계획
| 순서 | 작업 |
|------|------|
| 1 | git pull |
| 2 | 브랜치 생성: `feature/#N` |
| 3 | `/temp/TEMPLATE/context.md` 복사 → `/temp/#N/context.md` |
| 4 | 이슈 분석: 원인·영향 범위·수정 대상 파악 |
| 5 | 기획팀 계획서 작성 (context.md 계획서 섹션) |
| 6 | 기능 삭제 감지 시 → ⚠️ 경고 코멘트 준비 |

> ✅ GATE 1
> - [ ] /temp/#N/context.md 생성 완료
> - [ ] 계획서 작성 완료 (이슈 원인·영향 범위·작업 순서 명시)
> - 미통과 시 STEP 2 진행 금지

---

### STEP 2 — 코드 수정
| 순서 | 작업 |
|------|------|
| 1 | 계획서 기준 코드 수정 |
| 2 | 수정 중 결정사항·변경사항 즉시 context.md 수정 이력에 기록 |
| 3 | 기능 삭제 발생 시 ⚠️ 코멘트 삽입 |

> ✅ GATE 2
> - [ ] 코드 수정 완료
> - [ ] context.md 수정 이력 섹션 업데이트
> - 미통과 시 STEP 3 진행 금지

---

### STEP 3 — 품질 검사
| 순서 | 작업 |
|------|------|
| 1 | 린트·타입체크 자동 실행 |
| 2 | 이슈 테스트 20개 작성 |
| 3 | 단위 테스트 실행 |
| 4 | 전체 테스트 실행 (`--passWithNoTests`) |
| 5 | 커버리지 실행 (`--coverage`) → 80% 이상 확인 |
| 6 | unit-tests/ 실행 → 실패 시 사이드이펙트 수정 |

> ✅ GATE 3
> - [ ] 린트·타입체크 통과
> - [ ] 이슈 테스트 20개 통과
> - [ ] 커버리지 80% 이상
> - [ ] unit-tests/ 통과
> - [ ] context.md 테스트 결과 업데이트
> - 미통과 시 STEP 4 진행 금지

---

### STEP 4 — 사양서 판단
| 순서 | 작업 |
|------|------|
| 1 | 품질관리팀 코드 검토 |
| 2 | 사양서 업데이트 필요 여부 판단 |
| 3 | 판단 결과를 context.md 사양서 섹션에 기록 |

**판단 기준**
- 추가: 신규 기능·동작 변경 시
- 유지: 버그 수정만·동일 케이스 존재 시
- 삭제: 기능 삭제 시 항목 제거 + ⚠️ 코멘트

> ✅ GATE 4
> - [ ] 사양서 업데이트 판단 완료 (추가/유지/삭제 명시)
> - [ ] context.md 사양서 섹션 업데이트
> - 미통과 시 STEP 5 진행 금지

---

### STEP 5 — 보고서 & 마무리
| 순서 | 작업 |
|------|------|
| 1 | /reports/#N.md 저장 |
| 2 | gh issue comment 등록 (보고서 동일 내용) |
| 3 | 커밋: `git commit -m "... Closes #N"` |
| 4 | 푸시 → PR 생성 → 머지 |

> ✅ GATE 5
> - [ ] /reports/#N.md 저장 완료
> - [ ] gh issue comment 등록 완료
> - [ ] PR 머지 완료
> - 전체 완료 시 이슈 종료

## DB 규칙
- 작업 시작 전 반드시 Supabase 스키마 파일과 실제 테이블 구조를 먼저 확인한다
- `supabase/schema.sql`을 기준으로 컬럼명을 검증한다
- 확인 후 각 테이블 5행 샘플을 출력한다
- 코드 작성 시 컬럼명은 반드시 실제 스키마 기준으로 사용한다
- 스키마 확인 전에는 절대 코드 작성을 시작하지 않는다

## 파일 탐색 규칙
- Grep 우선 사용
- 전체 파일 읽기는 Grep 실패 시만 허용
- 한 번 읽은 파일 재읽기 금지
- 전체 읽기 시 이유 명시 필수

## 임시 파일 규칙
- 저장 위치: `/temp/#이슈번호/` (예: `/temp/86/analysis.md`)
- 프로젝트 루트·src 내 임시 파일 생성 금지
- .gitignore 등록 필수

## 보고서 형식
```
## 무엇을 발견
- 이슈 분석 결과·문제 원인

## 무엇을 수정
- 수정 파일·변경 요약

## 왜 그렇게 판단
- 결정 근거·대안 검토 내용

## 테스트
- [N]개 통과 / 커버리지 [N]%

## PR
- #[N] / 사양서: 추가 [N]개 또는 없음
```
